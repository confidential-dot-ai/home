# 2026-07-10 — `c8s luks` CLI for creating encrypted volumes

Stage 6 of the openbao integration. Same shape as the Stage 5 design doc:
lock the surface before coding, note the seam with everything already in
motion (Stage 5 injection, PR #295's guest-side scratch disk, PR #247's
in-flight workload adoption work), leave the code stubs to a follow-up.

## Goal

One command that goes from **nothing** to **workload pod annotation-ready**
for an openbao-gated LUKS volume:

    c8s luks create --workload api --name data --size 10Gi \
      --broker-secret secret/data/api/luks \
      --openbao-addr https://c8s-openbao.c8s-system.svc:8200 \
      --openbao-token-file ~/.config/c8s/bao-token

does:

1. Generate a strong random passphrase (32 bytes / hex, or an opaque token —
   see "Passphrase entropy" below).
2. Write it to openbao under `secret/data/<workload>/luks-<name>` in the
   `passphrase` field. The `secretBroker.releasePolicy` rule for the
   workload (`workloadId: <workload>, allow: [secret/data/<workload>/*]`)
   already covers it, so no policy change is required.
3. Provision the backing volume — pluggable: `--driver local` (hostPath
   under `/var/lib/c8s/luks/<workload>/<name>.img`, `losetup`, chmod-safe),
   `--driver pvc` (create a PVC in the workload's namespace and let the
   default StorageClass provision), `--driver csi` (hand off to a named
   StorageClass). Default: `pvc`.
4. Print (to stdout, YAML) the pod annotations + volume/volumeMount snippet
   the operator drops into the workload's PodSpec, ready to
   `kubectl apply` or pipe into a Helm values template. **No pod is
   started** — the CLI's job is provisioning + policy, not deploy.

Symmetric operations:

- `c8s luks list --workload <id>`  — enumerate volumes under
  `secret/data/<workload>/luks-*`, showing name, KV path, driver.
- `c8s luks destroy --workload <id> --name <name>` — delete the KV entry,
  then delete the volume (PVC/loop-file). Refuses if any pod is currently
  mounting it (kubectl `--all-namespaces` scan for the annotation).
- `c8s luks show --workload <id> --name <name>` — read-only view of the
  KV metadata + volume state, no passphrase disclosure. Useful for a
  human confirming a rotation didn't drop the record.

Deliberately out of scope for Stage 6:

- **Rotation.** `luks rotate` is a follow-up: add a new keyslot via
  `cryptsetup luksAddKey` on every pod that mounts the volume, then
  write the new KV version, then remove the old keyslot. Non-trivial
  because it needs to hit every open mapper. Deferred.
- **In-place format on first pod boot.** That's Stage 5's
  `mode=format-if-empty` — Stage 6 unconditionally sets `mode=open` in the
  emitted annotation because it guarantees the volume is already formatted
  by the CLI itself. Operators who prefer the lazy-format path pass
  `--defer-format` and get an emitted annotation with
  `mode=format-if-empty`; the CLI then only provisions the raw block
  device, does not `luksFormat`, and openbao holds the passphrase for
  first-mount to consume.

## Design

### The command tree

    c8s luks
      ├── create --workload --name --size [--driver pvc|local|csi]
      │           [--fstype ext4] [--broker-addr ...] [--openbao-addr ...]
      │           [--defer-format] [--passphrase-entropy 32] [--output yaml|json]
      ├── list --workload [--openbao-addr ...] [--output table|json|yaml]
      ├── destroy --workload --name [--force] [--openbao-addr ...]
      └── show --workload --name [--openbao-addr ...] [--output yaml|json]

All commands share a common flag set for openbao endpoint + auth
(`--openbao-addr`, `--openbao-token-file`, `--openbao-namespace`), a
`--broker-addr` for optional dry-run (compute the annotation without
touching openbao), and `--output` for machine-consumable formats.

### The provisioning drivers

Interface (`internal/cmds/luks/driver.go`):

    type Driver interface {
        // Provision allocates the backing block device and returns the
        // device path the eventual pod will see (e.g. /dev/loopN for
        // local, /dev/vdX for a CSI-managed PVC where the volume is
        // attached raw).
        Provision(ctx, ProvisionSpec) (Result, error)
        // Destroy releases the allocation.
        Destroy(ctx, DestroySpec) error
    }

    type ProvisionSpec struct {
        Workload, Name  string
        Size            resource.Quantity
        FSType          string
        DeferFormat     bool
        Passphrase      []byte  // used only when !DeferFormat
    }

    type Result struct {
        Device      string       // /dev/loop7, /dev/vdb, or ""
        Annotation  string       // "dev=/dev/loop7,mount=/data,secret=…"
        PodExtras   PodSnippet   // volume, volumeMount, resources to merge into workload spec
    }

Concrete drivers:

- `local` — for single-node dev clusters. `dd`+`losetup`+`chmod` on the
  operator host, `luksFormat` immediately with the freshly-generated
  passphrase, `mkfs.<fstype>` on the resulting mapper, `luksClose`, print
  the annotation. The emitted PodSpec adds a `hostPath` volume pointing
  at the loop file's underlying `.img` path plus a `securityContext:
  privileged: true` on the workload container (workaround for hostPath
  block devices in kata).
- `pvc` — creates a PVC with the requested size and the annotation
  `volume.confidential.ai/luks: workload=<id>,name=<name>` so a
  yet-to-be-written controller (deferred) can format on first bind. For
  now, `pvc + --defer-format` is the only combination we support because
  the CLI has no way to `luksFormat` a not-yet-bound PVC; document.
- `csi` — reserved for a per-vendor StorageClass path. Not implemented
  in this pass; the CLI errors with a "not implemented, use --driver pvc"
  message so the flag surface is stable when we add it.

### Passphrase entropy

Default: 32 bytes from `crypto/rand`, hex-encoded → 64 ASCII chars, well
above cryptsetup's minimum. Configurable via
`--passphrase-entropy <bytes>` (min 16, max 128). The passphrase itself
is opaque — cryptsetup doesn't care whether it's hex or base64 or
random-print-ASCII — hex is chosen because it's the least surprising in
JSON output.

We deliberately do NOT expose the passphrase to stdout even in
`--defer-format` mode: the CLI has already written it to openbao, and
letting it also print puts it in shell history. The only way to see the
passphrase is through openbao itself.

### Emitted annotation format

Exact match with Stage 5's parser:

    confidential.ai/luks-<name>: dev=<path>,mount=<mount>,secret=<vault-path>#passphrase,fstype=<fs>,mode=open

Plus the companion secrets annotation the parser cross-checks:

    confidential.ai/secret-<name>: <vault-path>#passphrase

`--output yaml` prints a partial PodSpec fragment:

    metadata:
      annotations:
        confidential.ai/luks-data: dev=/dev/vdb,mount=/data,secret=secret/data/api/luks-data#passphrase,fstype=ext4,mode=open
        confidential.ai/secret-data: secret/data/api/luks-data#passphrase

`--output json` returns the same, structured for a downstream YAML
composer. There is deliberately no "apply directly" flag — the CLI does
not know what workload to attach the annotation to, and picking one
implicit target is the kind of magic that surprises operators. Piping
into `kubectl edit` or Helm templates is the intended workflow.

### Interaction with kata + Stage 5's fail-closed guards

Stage 5's webhook rejects a `luks-<name>` annotation without matching
`secret-<name>` + `secrets-inject: true`. The CLI's `--output yaml`
therefore ALWAYS emits both annotations, and prints a warning if the
target workload doesn't already have `confidential.ai/secrets-inject:
"true"` set (which the CLI can only detect if the operator provides
`--pod-file` — a workload PodSpec to lint against).

### Auth

The operator invoking `c8s luks` needs write access to openbao. Two
supported modes:

- Root/manual token (`--openbao-token-file`) — for a dev cluster. Same
  auth path `scripts/secret-broker-demo.sh` uses.
- AppRole (`--openbao-approle-id --openbao-approle-secret-id-file`) —
  for a CI or automation account. The CLI performs `POST
  auth/approle/login` and uses the returned client token for the KV write.
  Symmetric with the broker's own AppRole path.

Both flow the same underlying openbao HTTP call at the end. This is not
about the broker's release-policy — the broker sees this write as any
other KV write; a stolen operator token means the volume can be
provisioned into openbao by an unauthorised operator, but that's a
standard secrets-management gap, orthogonal to the workload-side gate.

## Sequencing

1. **This design doc merged** (in the Stage 6 PR itself is fine — small
   change, one reviewer).
2. `c8s luks show` first — read-only, tests the openbao-client + policy
   plumbing without provisioning.
3. `c8s luks create --driver local --defer-format` — smallest slice that
   exercises annotation generation and openbao write.
4. `c8s luks create --driver local` full path (with format).
5. `c8s luks list` + `c8s luks destroy`.
6. `--driver pvc` path.
7. `--driver csi` deferred as a "returns not implemented" stub in this
   pass so the flag is stable.

## Follow-ups (tracked, not blocking Stage 6)

- **`c8s luks rotate`**: iterate mapper keyslots on every attached pod;
  requires talking to kubelet exec or introducing a per-node operator
  DaemonSet.
- **PVC first-bind format controller**: the yet-to-be-written controller
  the `pvc` driver's annotation targets. Alternative: fold the
  first-bind format into the workload pod's own init sequence
  (Stage 5's `mode=format-if-empty`) — nothing new needed, but the pod
  spec gets one more responsibility. Undecided; document the trade-off
  in the eventual PR.
- **PR #247's workload-adoption install path** may add a
  `--engine-workload-id` bit that inflects how `c8s luks create` picks
  its default `--workload`. Watch for that; probably a one-line default
  once #247 merges.
- **PR #295's scratch disk** is neighbouring, not overlapping — separate
  block device, guest-owned per-boot key, different threat model.
  Nothing to reconcile.

## Non-decisions

- **No new server binary.** All Stage 6 work lives in `cmd/c8s` and
  `internal/cmds/luks`. The operator does not learn about volumes at
  Stage 6.
- **No CRD.** The KV entry in openbao is the source of truth for the
  volume's existence; PVCs (when used) inherit their lifecycle from k8s.
  A `LUKSVolume` CRD would let us watch/reconcile drift but adds
  admission surface. Deferred.
